# DeFi 전략 실행용 멀티 에이전트 구조

이 문서는 `연 7~8% 안정 운용 전략(Option L2*)`을 실제 서비스로 구현/운영하기 위한 멀티 에이전트 조직도와 운영 규칙을 정의한다.

## 1) 목표

- 사용자 지갑(Phantom/Rabby 등) 연결 후 자산 예치
- 전략 규칙에 맞춰 체인/프로토콜별 자동 배분 및 리밸런싱
- 리스크 이벤트(디페그, 범위 이탈, 체인 이슈) 탐지와 대응
- 코드/보안/운영/성장(마케팅) 업무를 병렬 처리

## 2) 에이전트 구성

- `총괄 에이전트 (Chief-Orchestrator)`
- `기획 에이전트 (Strategy-PM)`
- `운용 에이전트 (Yield-Operator)`
- `개발 에이전트 (Protocol-Engineer)`
- `보안 에이전트 (Security-Guardian)`
- `마케팅 에이전트 (Growth-Marketer)`
- `디자이너 에이전트 (Design-Designer)`
- `컨설턴트 에이전트 (DeFi-Consultant)`

각 에이전트 상세 프롬프트는 `agents/` 폴더 참고.

## 3) 총괄 에이전트 제어 원칙

총괄은 아래 절차로 다른 에이전트를 제어한다.

1. **상태 수집**
   - TVL, 체인별 노출, 프로토콜별 노출, APY, 슬리피지, 경보 상태 수집
2. **우선순위 큐 생성**
   - `긴급(보안/손실 가능)` > `수익 영향` > `개발 생산성` > `성장 실험`
3. **작업 배정**
   - 기획: 정책/로드맵
   - 운용: 포지션 조정/리밸런싱
   - 개발: 기능 구현/버그 수정
   - 보안: 위협 평가/차단
   - 마케팅: 유입/전환 실험
   - 디자이너: UI/UX 개선, 정보 구조, 디자인 시스템
   - 컨설턴트: 프로토콜 위험도 검토, 파라미터 개선 권고
4. **게이트 승인**
   - 보안 승인 없이는 온체인 실행 금지
   - 기획 승인 없이는 전략 파라미터 변경 금지
5. **실행 후 평가**
   - KPI 비교(목표 대비 성과), 실패 원인, 다음 액션 자동 생성

## 4) 전략 가드레일 (PDF 반영)

- 단일 풀 최대 노출: `25%`
- 단일 체인 최대 노출: `60%`
- 최소 프로토콜 수: `3개 이상`
- 기본 리밸런싱: `분기 1회`
- ETH-USDC 집중 유동성: `±50% 범위` (상황별 조정)
- 비상 현금 버퍼: `5%`

## 5) 의사결정 RACI

- **총괄**: 최종 승인(A), 업무 배정(R)
- **기획**: 전략 정책 제안(R), 로드맵 정의(R)
- **운용**: 실행 계획/오더 생성(R)
- **개발**: 기능 코드 구현(R)
- **보안**: 위험 승인/차단(A)
- **마케팅**: GTM 실험(R)
- **디자이너**: UI/UX 설계 및 개선(R)
- **컨설턴트**: 프로토콜 검토 및 전략 권고(R)

## 6) 공통 KPI

- 연환산 수익률(CAGR): 목표 7~8%+
- Max Drawdown: -1% 이내(내부 경보 기준)
- 체인/프로토콜 집중도: 가드레일 위반 0건
- 보안 이벤트 MTTR: 30분 이내(심각도 High)
- 제품 KPI: 예치 전환율, 30일 잔존율, CAC 회수기간

## 7) 운영 루프

- **일간**: 리스크 모니터링, 이상치 경보
- **주간**: 전략 성과 리뷰, 백로그 재정렬
- **월간**: 제품/성장 실험 회고
- **분기**: 리밸런싱 + 파라미터 재평가

## 8) 즉시 실행 방법 (Cursor 기준)

1. `agents/` 각 파일 내용을 서브에이전트 시스템 프롬프트로 등록
2. 총괄 에이전트에만 의사결정 권한 부여
3. 다른 에이전트 출력은 총괄을 통해서만 실행 반영
4. PR/배포/온체인 실행 전 `Security-Guardian` 승인 필수

## 9) 주의

- 본 구조는 자동화 운영 설계 문서이며 투자자문이 아니다.
- 실제 자금 운용 전 테스트넷/소액 실험/감사를 선행해야 한다.

## 10) UI/UX 공통 준수 규칙 (전 에이전트 필수)

아래 규칙은 총괄 포함 모든 에이전트가 공통으로 준수한다.

- 텍스트 최소화: 불필요한 안내 문구/중복 타이틀 제거
- 행동 강조: 입금/인출 등 핵심 액션 버튼은 크고 명확하게 배치
- 블록체인 게이트: UI 지갑(Phantom·Solana)은 연결·서명으로 의사를 남기고, 실제 프로토콜 전송은 서버 실행 모드·RBAC에 따름(README «MVP 실행 범위»와 동일 선언 유지)
- 메뉴 단순화: 메인 메뉴는 핵심 여정 중심, 부가 기능은 `More`로 분류
- 상단 일관성: 좌측 로고(`Crypto8`) + 중앙 메인 메뉴 + 우측 지갑/도구 레이아웃 유지
- 확장 인터랙션: 지갑 상세는 클릭 시 아래로 펼치고 레이아웃 점프를 유발하지 않음
- 레이어 일관성: 상단바/드롭다운/모달 우선순위를 유지해 가림/먹통 방지
- 팝업 시야 기준: 실행 팝업은 본문 시야에서 바로 보이는 위치에 두고 상단 고착 배치는 피함
- 균형 배치: 상품 리스트와 실행 액션 영역 사이 간격을 확보해 시각 균형 유지
- 수치 표기: 누적 예치값 등 핵심 금액은 USD 기준으로 명시

## 11) 최신 운영 의도 요약 (2026-04)

아래 원칙은 총괄/디자이너/개발/컨설턴트 에이전트가 공통으로 우선 반영한다.

- 제품 경험은 "단순한 진입 + 명확한 실행 + 신뢰 가능한 확인"의 흐름으로 설계한다.
- 정보 구조는 핵심 여정과 보조 여정을 분리해 탐색 비용을 최소화한다.
- UI는 미니멀 텍스트 원칙을 따르되, 판단에 필요한 수치/맥락은 충분히 제공한다.
- 온체인 행위는 편의성보다 안전성을 우선하며 지갑 연동과 승인 증적을 보존한다.
- 외부 데이터 연동은 정확도와 함께 복원력을 고려해 다중 소스 전략으로 운영한다.

## 12) 제품 기술 경계 (MVP, 개발·보안·총괄 공유)

- **UI 지갑**: Solana(Phantom) 중심. 사용자 서명은 실행 의사 확인용이며, EVM 예치를 사용자 지갑이 직접 보내는 모델은 아님.
- **서버 실행**: `EXECUTION_MODE`·`LIVE_EXECUTION_CONFIRM` 조합이 실제 온체인 여부를 결정. 어댑터가 Aave/Uniswap/Orca를 담당.
- **예치 포지션 DB**: 생성·인출 API는 `orchestrator`만. `security`·`viewer`는 조회·승인·감사 역할에 집중.
- **입력 검증**: 예치 금액·`protocolMix` 크기·문자열 길이 상한을 API에서 강제해 남용·비정상 페이로드를 차단.
